## 문제 상황

바이낸스 선물 자동매매 봇을 운영하면서 세 가지 문제가 겹쳐서 드러났다.

1. 거래소 보호주문이 실패했다.

- 로그에는 `APIError(code=-4120): Order type not supported for this endpoint. Please use the Algo Order API endpoints instead.` 가 반복됐다.
- 봇 내부에는 `SL/TP` 값이 있었지만, 거래소에는 실제 조건부 주문이 안 걸리는 상태였다.

2. 수익권 피라미딩 후 손절/익절 라인이 유지되지 않았다.

- 예를 들어 `2트랜치` 추가 직후에는 새 평균단가 기준으로 `SL/TP`가 정상 계산됐다.
- 그런데 다음 사이클 `DB+거래소 매칭`이 돌면, 방금 올린 `SL/TP`가 다시 기본값으로 내려갔다.

3. 추가매수와 부분익절 시점이 Discord에 남지 않았다.

- 신규 진입/전량 청산 알림은 있었지만
- 피라미딩 추가, 부분익절은 실시간 복기가 어려웠다.

## 원인 분석

### 1. 보호주문 API 경로가 잘못됐다

기존 구현은 `STOP_MARKET`, `TAKE_PROFIT_MARKET`를 일반 `futures_create_order`로 보냈다.

하지만 최신 바이낸스 USDT-M 선물에서는 이런 조건부 주문을 일반 주문 엔드포인트가 아니라 `Algo Order` 엔드포인트로 보내야 했다. 그래서 `-4120`이 발생했다.

핵심 원인:

- 일반 주문 엔드포인트 사용
- 조건부 주문 타입과 엔드포인트가 맞지 않음

### 2. 피라미딩 후 SL/TP를 계산만 하고 유지하지 않았다

추가매수 직후 봇은 다음 값을 메모리에서만 갱신했다.

- 새 평균단가
- 새 손절가
- 새 익절가

그런데 그 다음 작업이 빠져 있었다.

- 갱신된 포지션 상태를 DB에 즉시 저장하지 않음
- 기존 거래소 보호주문 취소 후 새 `SL/TP`를 다시 걸지 않음

결과적으로 다음 `DB+거래소 매칭` 루프가 돌 때, 거래소 진입가를 기준으로 기본 `SL/TP`를 다시 계산해 덮어써 버렸다.

### 3. 알림 경로가 진입/청산에만 묶여 있었다

Discord 알림 함수는 이미 있었지만, 호출 지점이 아래 두 경우에 집중돼 있었다.

- 신규 진입
- 전량 청산

즉 피라미딩 전략에서 중요한 아래 이벤트는 누락돼 있었다.

- `2트랜치`, `3트랜치` 추가매수
- `트랜치 2/3` 도달 후 되돌림 기반 부분익절

## 해결 방법

### 1. 바이낸스 보호주문을 Algo Order 경로로 변경

`core/futures_client.py`에서 조건부 주문 생성 로직을 바꿨다.

- `STOP_MARKET`, `TAKE_PROFIT_MARKET`를 `algoOrder` 엔드포인트로 전송
- 구버전 `python-binance`에서 래퍼가 없을 수 있으므로 `_request_futures_api()`로 직접 호출
- 일반 미체결 주문뿐 아니라 조건부 주문도 함께 취소/조회되도록 보완

핵심 개념:

```python
order = self._create_algo_order(
    symbol=symbol,
    algoType="CONDITIONAL",
    side=close_side,
    type="STOP_MARKET",
    triggerPrice=formatted_price,
    quantity=formatted_qty,
    reduceOnly="true",
    workingType="MARK_PRICE",
    priceProtect="TRUE",
)
```

이후 실제 로그에서 아래처럼 보호주문이 정상 등록되는 것을 확인했다.

```text
[거래소 SL] SOLUSDT LONG qty=6.71 stop=87.2580 algoId=...
[거래소 TP] SOLUSDT LONG qty=6.71 tp=90.3585 algoId=...
```

### 2. 피라미딩 직후 새 SL/TP를 즉시 저장하고 재주문

`core/futures_trader.py`의 추가매수 로직을 수정했다.

추가된 동작:

- 새 평균단가 계산 직후 `self._persist_position(symbol)` 호출
- 기존 거래소 보호주문 전체 취소
- 새 수량/새 `SL/TP`로 보호주문 재등록

핵심 흐름:

```python
self._persist_position(symbol)
self.client.cancel_all_orders(symbol)
self.client.place_stop_loss(symbol, pos.side, pos.quantity, pos.stop_loss)
self.client.place_take_profit(symbol, pos.side, pos.quantity, pos.take_profit)
```

### 3. 동기화 루프에서 더 유리한 SL/TP를 보존

`DB+거래소 매칭` 로직도 손봤다.

기존에는 무조건:

- `entryPrice` 기준 기본 `SL`
- `entryPrice` 기준 기본 `TP`

로 다시 계산했다.

이제는:

- 방향이 바뀌었거나 값이 비정상이면 기본값으로 초기화
- 그 외에는 기존에 이미 끌어올린 `SL/TP`가 더 유리하면 그 값을 유지

예:

- `LONG`에서는 더 높은 `SL`, 더 높은 `TP` 유지
- `SHORT`에서는 더 낮은 `SL`, 더 낮은 `TP` 유지

### 4. Discord 알림 추가

`utils/discord_notify.py`에 아래 두 알림을 추가했다.

- `notify_position_scale()`
  - 몇 트랜치인지
  - 추가 수량
  - 총 수량
  - 새 평균단가
  - 새 `SL/TP`

- `notify_partial_profit()`
  - 부분익절 가격
  - 청산 수량
  - 남은 수량
  - 실현 손익
  - 익절 사유

이제 전략 복기 시 Discord 타임라인만 봐도 흐름이 바로 읽힌다.

## 결과

수정 후 확인된 개선 사항:

- `-4120` 보호주문 에러가 사라졌다.
- 부분익절 후 보호주문 재등록이 실제 거래소에 성공했다.
- 피라미딩 직후 계산된 `SL/TP`가 다음 분에 기본값으로 내려가는 버그를 제거했다.
- 추가매수와 부분익절 이벤트가 Discord에 남도록 했다.

특히 이번 이슈는 “계산 로직은 맞는데 상태 동기화가 틀리면 실전에서 완전히 다른 전략처럼 보인다”는 전형적인 사례였다.

## 배운 점

1. 실거래 봇에서는 “값을 계산했다”와 “그 값을 유지했다”는 전혀 다른 문제다.
2. 피라미딩 전략은 평균단가, 수량, 손절, 익절, 보호주문이 항상 한 덩어리로 같이 갱신돼야 한다.
3. 거래소 API는 주문 타입만 맞춰도 되는 게 아니라, 엔드포인트 계층까지 맞아야 한다.
4. 실전 운영에서는 Discord 같은 이벤트 로그가 단순 편의 기능이 아니라 복기 도구다.

다음 단계는 이 변경이 실시간 추가매수 구간에서도 계속 일관되게 유지되는지 모니터링하고, 필요하면 재시작 시점에도 보호주문을 강제로 재장전하는 방향까지 확장하는 것이다.
