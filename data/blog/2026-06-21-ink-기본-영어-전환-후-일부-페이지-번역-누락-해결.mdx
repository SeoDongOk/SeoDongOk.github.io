## 문제 상황

기본 언어를 영어로 변경한 뒤 구독, Sen 충전, 비자 관리, 글쓰기, 홈, 커뮤니티, 가이드 등 일부 화면에서 한국어·베트남어 문구나 번역 키 자체가 노출됐다. 지역 선택도 일부 화면에서 제한된 목록을 사용하고 있었다.

## 원인 분석

- 화면 컴포넌트에 사용자 노출 문구가 직접 작성되어 있었다.
- 영어 번역 사전에 동적으로 참조하는 홈·커뮤니티·가이드 키가 누락되어 있었다.
- 기존 코드가 `DEFAULT_LOCALE`을 베트남어라고 가정해 `locale === 'vi' || locale === DEFAULT_LOCALE`로 콘텐츠를 선택했다. 기본값을 영어로 바꾸자 영어에서도 베트남어 데이터가 선택됐다.
- 홈의 폴백 데이터는 번역 키를 실제 문구로 변환하지 않고 그대로 카드에 전달했다.
- 번역 함수가 변수 보간 인자를 받지 않아 날짜, 수량, 금액 관련 문구를 화면별로 직접 조합하고 있었다.

## 해결 방법

- 사용자 노출 문구를 `en`, `ko`, `vi` 사전으로 이동하고 세 사전의 키 구조를 동일하게 맞췄다.
- 번역 함수에 `{{variable}}` 보간을 추가하고 비자, 구독, 결제, 상대 시간 문구에 적용했다.
- 콘텐츠 선택 조건을 `locale === 'vi'`로 수정해 기본 영어가 베트남어로 잘못 분기되지 않도록 했다.
- 홈 폴백 데이터 생성 단계에서 번역 키를 현재 언어의 문구로 변환했다.
- 일자리·숙소 작성 화면에 공통 `RegionPicker`를 연결해 전국 시·도/시·군·구/읍·면·동 선택을 지원했다.
- 정적 분석으로 세 언어 사전의 누락·초과 키와 코드에서 사용하는 번역 키를 비교했다.

## 배운 점

기본 언어 변경은 상수 한 곳만 바꾸는 작업이 아니다. 기본 언어를 특정 언어의 폴백으로 간주한 분기, 동적 번역 키, API 콘텐츠 언어, 폴백 데이터 변환까지 함께 점검해야 한다. 번역 사전 구조 동등성 검사와 하드코딩 문자열 검사를 CI에 추가하면 같은 회귀를 조기에 막을 수 있다.
