## 들어가며

"개발팀이랑 금융쪽 팀원들이 같이 리뷰하는 게 좋을 거 같아."

바이낸스 선물 자동매매 봇(`sdo_auto_trader_crypto`)을 라즈베리파이에서 실거래로 돌리고 있는데, 설정을 어느 순간부터 혼자만 판단하기엔 버거워졌다. 그래서 ~/.claude/personas에 정리해둔 도메인 전문가 페르소나 5명을 불러 코드와 설정을 리뷰받았다.

모두 귀한 지적을 해줬지만, 오늘은 그중 **신하준의 피드백**에 집중해서 **실제로 코드에 반영한 내용**을 정리한다. 핵심은 하나다. **가격 시계열 위에서만 전략을 구성하던 걸 멈추고, 크립토 시장이 주는 "바깥 데이터"를 진입 결정에 끼워 넣는 것**.

## 신하준이 지적한 4가지

> "심볼 선택이랑 타임프레임 조합은 나쁘지 않은데, 크립토 시장 구조를 오해하고 있는 부분들이 보여."

### 1. SOL 단일 종목은 네트워크 리스크에 all-in

- Solana는 네트워크 중단 이력이 2021~2024년 사이 **7회 이상**
- 체인 정지 → 현물 급락 + 선물 funding 이상 현상
- 20x 레버리지 + 단일 종목 = 사실상 **Solana 가동 여부에 베팅**

→ 최소 **BTC + ETH + SOL 3종 분산**으로 상관은 있지만 비동조 이벤트에 대한 방어막 구축.

### 2. 15m 타임프레임에서 Funding 정산 시각은 독성 구간

- 바이낸스 USDT-M 선물 funding은 매 **8시간(UTC 00:00 / 08:00 / 16:00)** 정산
- 정산 직전 15분은 **hedging 수급 + funding 회피 거래**가 집중되어 가격이 구조적으로 흔들림
- 15m 전략은 이 구간 진입 확률이 1/32 (= 15분 / 8시간). 작아 보여도 누적 손실이 유의미

### 3. Funding Rate 자체가 방향성 편향 신호

- 평균 ±0.01%/8h, 극단 ±0.1%/8h (연 환산 ±3~109%)
- 한쪽으로 과밀된 funding은 **반대 방향 스퀴즈의 선행 지표**
- 롱 funding 과열 → 숏 스퀴즈, 숏 funding 과열 → 롱 스퀴즈

### 4. 온체인 지표 = 아예 안 쓰고 있었음

> "8개 지표 조합하는 건 좋은데, 다 가격 기반이네. Long/Short Ratio 하나만 봐도 edge 생겨."

15m에서 RSI와 Ichimoku는 사실상 시그널이 아니라 **노이즈**에 가깝다는 지적도 함께. 크립토 강세 구간에서 RSI 70+로 몇 주간 추가 상승하는 케이스가 흔하니까.

## 적용: 3개의 온체인/시간 필터

### 필터 1 — Funding 정산 시간 차단 (`utils/time_filter.py`)

UTC 00:00 / 08:00 / 16:00 **전 15분 ~ 후 5분** 구간을 진입 금지로 설정.

```python
from datetime import datetime, timezone

FUNDING_HOURS_UTC = (0, 8, 16)

def is_funding_window(
    *, buffer_before_min: int = 15, buffer_after_min: int = 5,
    now: datetime | None = None,
) -> bool:
    t = now or datetime.now(timezone.utc)
    current_min = t.hour * 60 + t.minute
    for funding_hour in FUNDING_HOURS_UTC:
        funding_min = funding_hour * 60
        start = (funding_min - buffer_before_min) % 1440
        end = (funding_min + buffer_after_min) % 1440
        if start <= end:
            if start <= current_min <= end:
                return True
        else:
            # 자정 넘어가는 구간 (23:45 ~ 00:05)
            if current_min >= start or current_min <= end:
                return True
    return False
```

자정 경계를 넘나드는 구간(예: 23:45~00:05) 처리를 깜빡하면 이 구간에 오히려 "언제나 통과"가 되니 주의.

### 필터 2 — Funding Rate 극단 차단 (`utils/funding_filter.py`)

Binance의 무료 엔드포인트 `GET /fapi/v1/premiumIndex`로 현재 funding rate를 받아와서 절댓값이 0.05% 넘으면 차단.

```python
import requests, time

_PREMIUM_INDEX_URL = "https://fapi.binance.com/fapi/v1/premiumIndex"
_cache: dict[str, tuple[float, float]] = {}
_CACHE_TTL_SEC = 60  # 1분 캐싱 (API 쿼터 보호)

def get_funding_rate(symbol: str, *, timeout: float = 3.0) -> float | None:
    cached = _cache.get(symbol)
    now = time.time()
    if cached is not None and (now - cached[1]) < _CACHE_TTL_SEC:
        return cached[0]
    try:
        resp = requests.get(_PREMIUM_INDEX_URL, params={"symbol": symbol}, timeout=timeout)
        resp.raise_for_status()
        data = resp.json()
        rate = float(data.get("lastFundingRate", 0))
        _cache[symbol] = (rate, now)
        return rate
    except (requests.RequestException, ValueError, KeyError) as e:
        return None  # 조회 실패 시 차단 안 함 (관대한 기본값)

def is_funding_extreme(symbol: str, *, threshold_pct: float = 0.05) -> bool:
    rate = get_funding_rate(symbol)
    if rate is None:
        return False
    return abs(rate) > (threshold_pct / 100)
```

- **캐싱 필수**: funding은 8시간마다 바뀌는 값이라 매번 HTTP 요청 돌릴 이유가 없음. 60초 TTL.
- **실패 시 관대**: API 장애로 차단되면 거래 기회 전부 날아감. 조회 실패는 `False`(차단 안 함)로.

### 필터 3 — Long/Short Ratio 스퀴즈 회피 (`utils/onchain_filter.py`)

`GET /futures/data/globalLongShortAccountRatio`로 15분 주기 집계된 계정 기준 롱/숏 비율을 조회. **2.0 초과(롱 과밀) 또는 0.5 미만(숏 과밀)**이면 차단.

```python
_LONGSHORT_URL = "https://fapi.binance.com/futures/data/globalLongShortAccountRatio"
_cache: dict[str, tuple[float, float]] = {}
_CACHE_TTL_SEC = 180  # 15m 갱신 데이터라 3분 캐싱

def get_long_short_ratio(
    symbol: str, *, period: str = "15m", timeout: float = 3.0
) -> float | None:
    cache_key = f"{symbol}:{period}"
    cached = _cache.get(cache_key)
    now = time.time()
    if cached is not None and (now - cached[1]) < _CACHE_TTL_SEC:
        return cached[0]
    try:
        resp = requests.get(
            _LONGSHORT_URL,
            params={"symbol": symbol, "period": period, "limit": 1},
            timeout=timeout,
        )
        resp.raise_for_status()
        data = resp.json()
        if not data:
            return None
        ratio = float(data[0].get("longShortRatio", 0))
        _cache[cache_key] = (ratio, now)
        return ratio
    except (requests.RequestException, ValueError, KeyError, IndexError):
        return None

def is_squeeze_risk(
    symbol: str, *, upper: float = 2.0, lower: float = 0.5, period: str = "15m",
) -> tuple[bool, float | None]:
    ratio = get_long_short_ratio(symbol, period=period)
    if ratio is None:
        return False, None
    if ratio > upper:
        return True, ratio  # 롱 과밀 → 숏 스퀴즈 위험
    if ratio < lower:
        return True, ratio  # 숏 과밀 → 롱 스퀴즈 위험
    return False, ratio
```

**주의**: Binance가 제공하는 Long/Short 지표는 3종류가 있다.

- `globalLongShortAccountRatio`: 전체 계정 수 기준 (지금 이거)
- `topLongShortAccountRatio`: 상위 트레이더 **계정 수** 기준
- `topLongShortPositionRatio`: 상위 트레이더 **포지션 규모** 기준

소액 개인 투자자들의 심리를 보려면 `global`, 고래의 방향을 보려면 `top...Position`이 더 유용하다. 지금은 "과밀 스퀴즈 위험"을 본다는 맥락이라 global을 선택했다. 향후 고래 역행 전략으로 확장할 때는 top을 추가할 예정.

### 필터 통합 — `FuturesTrader._can_open_new_position`

기존 중복/한도/일일 손실 체크 뒤에 3개 필터를 순차 적용:

```python
# ── 신하준 피드백 필터 1: Funding 정산 시간대 ───────────
if self.settings.time_filter_enabled and is_funding_window(
    buffer_before_min=self.settings.time_filter_buffer_before_min,
    buffer_after_min=self.settings.time_filter_buffer_after_min,
):
    logger.info(f"{symbol}: Funding 정산 시간대 — 진입 차단")
    return False

# ── 신하준 피드백 필터 2: Funding Rate 극단 ─────────────
if self.settings.funding_filter_enabled and is_funding_extreme(
    symbol, threshold_pct=self.settings.funding_rate_threshold_pct
):
    return False

# ── 신하준 피드백 필터 3: Long/Short Ratio 스퀴즈 ───────
if self.settings.longshort_filter_enabled:
    is_risk, _ratio = is_squeeze_risk(
        symbol,
        upper=self.settings.longshort_upper,
        lower=self.settings.longshort_lower,
        period=self.settings.longshort_period,
    )
    if is_risk:
        return False
```

## 배운 점 / 자주 하는 실수

### 온체인 데이터의 "무료"는 생각보다 많다

Binance만 해도 인증 없이:

- `premiumIndex` — funding rate, mark price
- `globalLongShortAccountRatio` — 전체 롱/숏 계정 비율
- `topLongShortPositionRatio` — 고래 포지션 비율
- `openInterest`, `openInterestHist` — 미결제 약정
- `takerlongshortRatio` — 테이커 매수/매도 비율

**"온체인 = Dune/Glassnode 유료"라고 생각했는데 사실 차익거래 시그널의 70%는 거래소가 공짜로 준다.** 이거 모르는 게 edge 손해.

### 캐싱 전략이 핵심

1분짜리 실시간 전략에 매 사이클 HTTP 요청 3번씩 넣으면 분당 API call 180회(3심볼 × 3필터 × 20체크). Binance IP weight 한도 내지만 굳이 부담 줄 필요 없음. **funding은 60초 TTL, L/S는 180초 TTL**로 조절.

### 실패 기본값은 "통과" 쪽으로

필터 조회 실패 = 차단, 으로 하면 API 장애 한 번에 하루 거래 전부 날아감. 반대로 **실패 = 통과**로 두고 에러 로그만 남기는 게 현실적. 원칙적으로는 "보수적"이 옳지만 수익이 목적인 봇에서는 가용성도 중요한 trade-off다.

### 리뷰 페르소나는 예상보다 쓸모 있다

5명이 각각 다른 층위를 본다.

- 리스크 매니저는 "SL 15% + 레버 20x = 실제 가격 0.75%"라는 R:R 역전 구조를 잡아냈고,
- 퀀트 개발자는 "합산 임계값 3.0이 전체 가중치의 28.6%라는 걸 문서에 안 쓴 걸" 지적했고,
- 크립토 전략가는 "15m에서 Ichimoku는 빡빡하다"는 **도메인 감각**을 줬고,
- 백엔드 엔지니어는 "SL/TP를 거래소에 주문으로 안 걸고 1분 폴링만 하면 봇 다운 시 청산 리스크 그대로"라는 **운영 관점 버그**를 짚었고,
- DevOps는 "monitoring 0, heartbeat 없음"을 체크했다.

**한 명이 모든 걸 다 볼 수는 없다.** 자동매매는 특히 그렇다.
