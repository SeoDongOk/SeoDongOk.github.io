## 문제 상황

INK 프로젝트에서 에이전트에게 한 번만 수정을 맡기면, 구현은 끝났다고 해도 `lint`, `test`, `build`가 마지막에 깨지는 경우가 자주 남았다.

특히 React Native 쪽은 기능 하나를 붙여도 아래처럼 후속 검증이 계속 필요했다.

- 소셜 로그인 UI 추가 후 `eslint` 확인
- 채팅 기능 추가 후 `jest` 확인
- Supabase 연동 후 타입/런타임 경로 확인

그래서 목표는 단순했다. 에이전트가 한 번 수정하고 끝나는 대신, 검증에 실패하면 그 로그를 다시 읽고 다음 수정까지 이어서 반복하게 만드는 `Ralph Loop`를 붙이는 것이었다.

## 원인 분석

문제는 “자동 수정 루프” 자체보다도, 이를 실제 CLI 환경에 맞게 안정적으로 감싸는 부분에 있었다.

구조적으로 필요한 요소는 아래 네 가지였다.

1. 작업 프롬프트를 에이전트에게 전달
2. 바깥에서 `test`, `lint`, `typecheck`, `build` 같은 검증 실행
3. 실패 로그를 다음 iteration 프롬프트에 다시 주입
4. 성공하거나 `maxIterations`를 넘길 때까지 반복

하지만 실제로 구현해보니 두 가지 현실적인 문제가 있었다.

- `codex exec`는 최상위 CLI와 옵션 호환이 완전히 같지 않았다.
- 중첩된 CLI 실행 환경에서는 권한/세션 제약 때문에 루프가 기대처럼 끝까지 자동화되지 않는 경우가 있었다.

즉, Ralph Loop의 핵심은 맞았지만 “에이전트 래퍼 + 검증 래퍼 + 실행 환경” 세 층을 같이 맞춰야 했다.

## 해결 방법

우선 공통 실행기와 설정 파일을 따로 분리했다.

- 실행기: `scripts/ralph-loop.mjs`
- 설정: `ralph-loop.config.json`
- 사용 문서: `docs/ralph-loop.md`

설정은 에이전트와 프로젝트 프로필을 분리하는 형태로 잡았다.

```json
{
  "defaults": {
    "maxIterations": 5,
    "agentTimeoutMin": 20,
    "checkTimeoutMin": 10
  },
  "agents": {
    "codex": {
      "command": "codex",
      "args": ["exec", "--skip-git-repo-check", "--ephemeral", "--sandbox", "workspace-write"]
    },
    "claude": {
      "command": "claude",
      "args": ["-p", "--permission-mode", "acceptEdits"]
    }
  }
}
```

React Native와 Web에는 각각 다른 검증 세트를 연결했다.

```json
{
  "profiles": {
    "ink-rn": {
      "cwd": "./vietnam-in-korea-superapp-rn",
      "verify": ["npm test -- --watch=false", "npm run lint"]
    },
    "ink-web": {
      "cwd": "./vietnam-in-korea-superapp-web",
      "verify": ["npm run lint", "npm run typecheck", "npm run build"]
    }
  }
}
```

실행 예시는 아래처럼 단순하게 유지했다.

```bash
cd /Users/seodong-og/project_vs/plan

node scripts/ralph-loop.mjs \
  --agent codex \
  --profile ink-rn \
  --task "Apple 로그인 구현"
```

그리고 실제 작업에서는 완전 자동화가 깨지는 구간이 있더라도, 같은 원칙으로 수동 반복을 이어갔다.

1. 기능 구현
2. `npm test`, `eslint`, `tsc` 같은 검증 실행
3. 실패 원인 확인
4. 다시 수정

예를 들어 채팅 기능 작업에서는 `App.test.tsx`가 네이티브 모듈 때문에 깨졌고, 이 문제를 mock으로 복구한 뒤 다시 루프를 진행했다. 이때 생긴 세부 이슈는 별도 트러블슈팅 포스트로 남겼다.

## 배운 점

- Ralph Loop는 “AI가 알아서 고친다”가 아니라, **검증 기준을 기계적으로 명확하게 만들어야만** 효과가 난다.
- `lint`, `test`, `typecheck`, `build`처럼 종료 코드가 분명한 명령만 붙이는 편이 좋다.
- CLI 자동화는 프롬프트보다 실행 환경 제약에서 더 자주 깨진다. 특히 서브커맨드 옵션 차이와 세션 권한 문제가 핵심이다.
- 완전 자동화가 막히더라도, `구현 -> 검증 -> 실패 로그 기반 재수정`의 작업 방식 자체는 그대로 유지할 수 있다.
- 결국 Ralph Loop의 본질은 스크립트 이름이 아니라, **수정 결과를 검증으로 닫는 개발 루프를 강제하는 것**이다.
