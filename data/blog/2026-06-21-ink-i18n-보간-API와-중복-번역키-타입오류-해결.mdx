## 문제 상황

화면의 하드코딩 문구를 번역 키로 교체하면서 `t(key, params)` 형태를 사용했지만 기존 `t` 함수는 인자를 하나만 허용해 `Expected 1 arguments, but got 2` 오류가 발생했다. 한국어와 베트남어 사전에는 기존 `home.askAi`, `home.jobs`와 같은 키를 다시 추가해 중복 프로퍼티 오류도 발생했다.

## 원인 분석

기존 i18n 구현은 문자열 조회만 지원하고 `{{name}}` 형태의 변수 보간을 제공하지 않았다. 또한 로케일 파일마다 같은 의미의 키가 이미 다른 위치에 존재했는데 새 객체를 추가하면서 중복 키가 만들어졌다.

## 해결 방법

`getMessage`와 LocaleContext의 `t` 시그니처에 선택적 파라미터 객체를 추가하고, `{{key}}` 토큰을 치환하는 공통 보간 함수를 구현했다. 중복된 번역 키는 제거하고 기존 객체에 필요한 하위 키만 병합했다. 이후 TypeScript 타입 검사를 다시 실행해 오류가 사라진 것을 확인했다.

## 배운 점

다국어 전환에서는 단순 문자열뿐 아니라 숫자, 날짜, 상품명처럼 동적 값이 포함된 문구가 많다. 초기 i18n API부터 타입이 있는 보간 기능을 제공해야 화면별 수동 `replace`와 하드코딩을 줄일 수 있다. 새 번역 키를 추가하기 전에는 기존 사전의 전체 키 경로를 검사해 중복을 방지해야 한다.
