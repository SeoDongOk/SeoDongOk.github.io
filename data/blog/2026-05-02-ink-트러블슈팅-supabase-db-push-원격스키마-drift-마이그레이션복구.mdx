## 문제 상황

구독 서비스용 새 마이그레이션을 배포하려고 `supabase db push`를 실행했는데, 정작 새 파일이 아니라 예전 미적용 마이그레이션에서 계속 실패했다.

처음 실패한 에러는 이런 형태였다.

```text
ERROR: column hl.room_type does not exist
```

수정 후 다시 밀자 다음엔 이런 식으로 또 깨졌다.

```text
ERROR: operator does not exist: community_region = text
```

즉, 새 구독 스키마 문제라기보다 **원격 DB가 로컬 마이그레이션이 가정한 상태와 달라져 있었던 것**이 핵심이었다.

## 원인 분석

원격 프로젝트에는 아래처럼 2026-04-11 이후 마이그레이션이 통째로 미적용 상태였다.

- `20260411000001`
- `20260411000002`
- `20260411000003`
- `20260411000004`
- `20260427000001`
- `20260427000002`
- `20260502000003`
- `20260502000004`

그런데 원격 DB는 그 사이에 일부 테이블/컬럼이 수동 또는 다른 경로로 이미 바뀌어 있었다.

그래서 예전 마이그레이션이 기대하던 것과 실제 원격 상태가 달랐다.

- `housing_listings` 일부 컬럼 누락
- `region` 컬럼 타입이 text가 아니라 enum

결국 **미적용 마이그레이션을 지금 시점의 원격 스키마에 맞게 방어적으로 바꿔야 했다**.

## 해결 방법

문제가 된 `20260411000001_ink_bookmarks_ai_views.sql`를 수정했다.

### 1. `v_housing_feed`를 컬럼 유무에 따라 동적으로 생성

기존에는 아래처럼 컬럼이 있다고 가정했다.

```sql
select
  hl.room_type,
  hl.deposit,
  hl.monthly_rent,
  hl.price_text
```

수정 후에는 `information_schema.columns`를 조회해서 컬럼이 있으면 실제 컬럼을, 없으면 `null::text` / `null::int`를 쓰도록 바꿨다.

즉, 부분 생성된 원격 테이블에서도 뷰 생성이 실패하지 않게 만들었다.

### 2. `v_home_metrics`에서 enum/text 비교를 캐스팅으로 통일

원격에서는 `region` 타입이 text가 아니어서 비교가 깨졌기 때문에 아래처럼 바꿨다.

```sql
j.region::text = r.region
h.region::text = r.region
n.region::text = r.region
```

이렇게 하면 지역 타입이 enum이든 text든 뷰 레벨에서는 비교가 가능하다.

### 3. 다시 `supabase db push`

수정 후 다시 push했고, 이번에는 기존 pending 마이그레이션과 새 구독 스키마까지 모두 적용됐다.

## 배운 점

- 새 마이그레이션이 실패한다고 해서 항상 “새 파일 문제”는 아니다. 원격에 남아 있던 **예전 pending migration**이 먼저 터질 수 있다.
- Supabase 프로젝트를 오래 운영하면 원격 스키마 drift가 생기기 쉽다. 특히 view는 컬럼 하나만 달라도 바로 깨진다.
- 운영 중인 원격 DB에 뒤늦게 예전 마이그레이션을 적용해야 할 때는, 이상적인 초기 상태를 가정하지 말고 **현재 상태를 흡수하는 방어적 SQL**이 필요하다.
- `migration list`로 로컬/원격 차이를 먼저 보는 습관이 중요하다.
