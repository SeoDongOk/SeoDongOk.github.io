## 문제 상황

INK 앱에 계정 삭제 기능을 붙인 뒤 다시 흐름을 점검해보니, 기존 `delete-account` 함수는 아래 버킷만 정리하고 있었다.

- `avatars`
- `profile-photos`
- `private-albums`
- `community-photos`

그런데 이후 채팅 기능을 추가하면서 `chat-images` 버킷이 새로 생겼고, 사진 메시지 업로드도 이 버킷을 사용하게 됐다. 이 상태로는 사용자가 탈퇴해도 채팅 이미지 파일이 스토리지에 남을 수 있었다.

## 원인 분석

원인은 기능 추가 순서 때문이었다.

1. 먼저 계정 삭제 함수를 구현
2. 이후 채팅 이미지 업로드 기능 추가
3. `delete-account`의 정리 대상 버킷 목록은 갱신하지 않음

즉, 계정 삭제 로직 자체가 잘못된 것이 아니라 **새 스토리지 자산이 생겼는데 삭제 루틴이 따라 업데이트되지 않은 것**이었다.

## 해결 방법

`supabase/functions/delete-account/index.ts`의 버킷 목록에 `chat-images`를 추가했다.

```ts
const USER_BUCKETS = [
  'avatars',
  'profile-photos',
  'private-albums',
  'community-photos',
  'chat-images',
]
```

채팅 업로드 경로도 함께 확인했다. 현재 채팅 이미지는 아래 규칙을 따른다.

```ts
const filePath = `${userId}/${Date.now()}.${extension}`
```

즉, 계정 삭제 시 `userId` prefix 기준으로 파일을 지우는 기존 방식과 맞아 떨어진다.

추가로 배포/검증용 체크리스트도 별도 문서로 정리해 두었다.

## 배운 점

- 계정 삭제 기능은 “테이블 cascade”만 보면 끝이 아니다. Storage 버킷도 같은 수준으로 관리해야 한다.
- 새 업로드 기능을 추가할 때는 반드시 “탈퇴 시 정리 경로”도 같이 점검해야 한다.
- 사용자 생성/수정/삭제처럼 수명주기 전체를 다루는 기능은, 나중에 붙는 서브 기능이 누락되기 쉬우므로 체크리스트 문서가 유용하다.
