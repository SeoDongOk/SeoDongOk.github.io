## 문제 상황

iOS MVP 기준으로 로그인 수단을 `Apple`, `Google`, `이메일`만 남기고 `Kakao`, `Zalo`, `Phone`은 로그인 화면에서 숨겼다.

그런데 앱을 다시 띄우자 Metro가 아래 에러로 중단됐다.

```text
Unable to resolve module @react-native-seoul/kakao-login
```

로그인 화면에서는 Kakao 버튼을 이미 제거했는데도, Metro는 여전히 `kakaoAuth.ts`를 따라가고 있었다.

## 원인 분석

문제는 UI 노출과 번들 import가 서로 다른 레이어라는 점이었다.

- `LoginScreen.tsx`에서는 Kakao 버튼을 제거
- 하지만 `AuthContext.tsx`는 여전히 `signInWithKakao`, `signInWithLine`, `signInWithZalo`, `signInWithFacebook`를 정적으로 import

Metro는 화면에서 그 경로를 쓰지 않더라도, 정적 import가 남아 있으면 모듈 그래프를 따라가며 해석을 시도한다.

즉 “버튼만 숨김”으로는 충분하지 않았고, iOS MVP에서 제외한 로그인 provider의 import 자체를 끊어야 했다.

## 해결 방법

`AuthContext.tsx`에서 테스트 단계 provider import와 분기를 제거했다.

```ts
import { supabaseClient } from '../services/supabase/client'
```

그리고 `loginWithProvider`, `signUpWithProvider`는 iOS MVP에서 허용한 provider만 처리하도록 바꿨다.

```ts
if (provider === 'google' || provider === 'apple') {
  await loginWithOAuth(provider, locale)
  return
}

throw new Error(`Unsupported login provider for iOS MVP: ${provider}`)
```

이후 검증:

- `npm run typecheck` 통과
- 변경 파일 `eslint` 에러 0
- Metro에서 더 이상 `@react-native-seoul/kakao-login` 해석을 시도하지 않음

## 배운 점

MVP 범위를 줄일 때는 “화면에서 안 보인다”와 “번들에서 안 읽힌다”를 분리해서 봐야 한다.

특히 React Native/Metro에서는 아래 순서로 줄이는 편이 안전하다.

1. UI 버튼 제거
2. Context/Service import 제거
3. provider 분기 제거
4. 필요하면 네이티브 패키지 정리

이번 경우처럼 1단계만 하고 멈추면, 런타임에서는 안 보여도 빌드/번들 단계에서 바로 다시 터질 수 있다.
