# 프로젝트 개선 제안서

## 개요

이 프로젝트는 Next.js App Router, Contentlayer, MDX 기반으로 구성된 개인 블로그/포트폴리오 사이트입니다.  
기본 구조는 안정적이고 글 작성 중심의 운영에는 적합합니다. 다만 현재 상태는 스타터 템플릿의 기본 설정과 예시 데이터가 일부 남아 있어, 실제 개인 사이트로서의 완성도와 신뢰도를 높이기 위한 정리가 필요합니다.

이 문서는 현재 코드베이스를 기준으로 우선순위가 높은 개선 사항을 정리한 문서입니다.

## 가장 먼저 개선할 항목

### 1. 사이트 메타데이터 실제 정보로 교체

현재 사이트 정보에는 템플릿 기본값과 placeholder가 남아 있습니다.

- `siteUrl`이 실제 개인 도메인이 아니라 템플릿 도메인으로 설정되어 있음
- `siteRepo`가 본인 저장소가 아니라 원본 템플릿 저장소를 가리킴
- 이메일, GitHub, LinkedIn, Medium 등 소셜 링크가 placeholder 상태임

영향 범위:

- SEO
- Open Graph
- Twitter 카드
- RSS
- sitemap.xml
- robots.txt

관련 파일:

- `data/siteMetadata.js`
- `app/layout.tsx`
- `app/sitemap.ts`
- `app/robots.ts`

권장 조치:

- 실제 배포 도메인 반영
- 실제 GitHub 저장소 주소 반영
- 사용하지 않는 SNS 링크 제거
- 사용하는 채널만 남기고 정확한 URL로 교체

### 2. 프로젝트 페이지 예시 데이터 제거

현재 프로젝트 목록에는 실제 작업물이 아니라 템플릿 예시 데이터가 남아 있습니다.

- Google 예시
- Time Machine 예시

이 상태는 방문자에게 사이트가 미완성처럼 보이게 만듭니다.

관련 파일:

- `data/projectsData.ts`
- `app/projects/page.tsx`

권장 조치:

- 실제 프로젝트 2~4개로 교체
- 각 프로젝트에 아래 정보 포함
  - 프로젝트 목적
  - 사용 기술
  - 본인이 맡은 역할
  - 결과 또는 성과
  - GitHub / Demo 링크

### 3. 작성자 정보 정리

작성자 정보에 잘못된 링크가 포함되어 있습니다.

- LinkedIn 필드에 public-profile settings URL이 들어감
- GitHub 필드에 GitHub가 아니라 LinkedIn 주소가 들어감
- 본문 소개가 `Hi` 한 줄로 끝남

관련 파일:

- `data/authors/default.mdx`

권장 조치:

- GitHub, LinkedIn URL 정상화
- 자기소개 보강
- 현재 하는 일, 관심 분야, 기술 스택, 블로그 주제 명시

예시 방향:

- 퀀트 트레이딩 시스템 개발
- 자동매매 실험 기록
- 백엔드 / 데이터 / 인프라 경험
- 시행착오와 성능 개선 기록

### 4. 빌드 안정성 개선

현재 빌드 시 Google Fonts 네트워크 호출에 의존합니다.  
검토 환경에서는 `fonts.googleapis.com` 접근 실패로 `npm run build`가 실패했습니다.

관련 파일:

- `app/layout.tsx`

현재 문제:

- `next/font/google` 사용
- 네트워크 상태나 CI 환경에 따라 빌드 실패 가능

권장 조치:

- `next/font/local`로 전환
- 또는 폰트 의존성을 줄이고 로컬 폰트 에셋으로 관리

이 항목은 배포 안정성과 직접 연결되므로 우선순위가 높습니다.

## 품질을 높이는 개선 항목

### 5. 홈 화면 메시지 강화

현재 홈 화면 헤드라인은 `Latest`로만 표시됩니다.  
이 표현만으로는 방문자가 이 사이트의 주제를 빠르게 이해하기 어렵습니다.

관련 파일:

- `app/Main.tsx`

권장 조치:

- 메인 카피를 사이트 주제 중심으로 변경

예시:

- 퀀트 트레이딩 시스템 개발 기록
- 자동매매, 백테스트, 실험 로그를 정리하는 기술 블로그
- Full-stack와 Quant를 연결하는 개발 노트

### 6. 블로그 글의 완성도 향상

일부 글은 아직 템플릿형 일지 구조에 가깝습니다.

예:

- 변경 없음
- 오늘 배운 것 작성 placeholder

관련 파일 예시:

- `data/blog/2026-03-27-quant-dev-log.mdx`

권장 조치:

일일 로그 형식을 유지하더라도 다음 구조를 권장합니다.

1. 오늘 작업한 내용
2. 왜 이 작업을 했는지
3. 결과가 어땠는지
4. 막힌 점이 무엇인지
5. 다음 액션이 무엇인지

이렇게 정리하면 단순 기록에서 끝나지 않고, 누적될수록 개발 아카이브 가치가 올라갑니다.

### 7. 템플릿 흔적 제거

현재 사이트와 저장소에는 원본 스타터 템플릿의 흔적이 남아 있습니다.

- Footer에 원본 테마 링크 노출
- README가 거의 템플릿 원문 상태

관련 파일:

- `components/Footer.tsx`
- `README.md`

권장 조치:

- Footer 문구를 개인 브랜드 기준으로 변경
- README를 이 프로젝트 전용 문서로 다시 작성

README에 들어가면 좋은 내용:

- 프로젝트 소개
- 기술 스택
- 로컬 실행 방법
- 글 작성 방법
- 배포 방법
- 디렉터리 구조

### 8. 패키지 매니저 정리

현재 저장소에는 `package-lock.json`과 `yarn.lock`이 함께 존재합니다.  
하지만 `package.json`에는 Yarn 사용이 명시되어 있습니다.

관련 파일:

- `package.json`
- `package-lock.json`
- `yarn.lock`

권장 조치:

- npm 또는 yarn 중 하나만 유지
- 사용하지 않는 lockfile 제거

혼용 상태는 팀 작업이나 배포 환경에서 의존성 차이를 만들 수 있습니다.

### 9. 소셜 아이콘 클래스 처리 개선

소셜 아이콘 크기 클래스가 동적 문자열로 구성되어 있습니다.

관련 파일:

- `components/social-icons/index.tsx`

현재 코드 방식:

- `h-${size} w-${size}`

문제 가능성:

- Tailwind가 동적 문자열 클래스를 정적으로 수집하지 못할 수 있음
- 프로덕션 빌드에서 특정 클래스가 누락될 위험이 있음

권장 조치:

- 크기별 class map 사용
- 또는 inline style 사용

## 중장기 개선 제안

### 10. 프로젝트 브랜딩 강화

현재 구조는 블로그 기능은 충분하지만, 방문자 입장에서 "무엇을 잘하는 사람인지"가 첫 화면에서 강하게 드러나지 않습니다.

권장 방향:

- 홈 상단에 핵심 소개 섹션 추가
- 관심 분야를 3개 정도로 요약
  - Quant Trading
  - Backend Engineering
  - Automation / Data
- 대표 프로젝트 또는 대표 글을 상단에 고정 노출

### 11. 카테고리/시리즈 구조 보강

현재는 태그 중심 구조입니다.  
콘텐츠가 계속 쌓일 예정이라면 아래와 같은 시리즈 구조를 추천합니다.

- 퀀트 개발 일지
- 백테스트 분석
- 자동매매 시스템 설계
- 인프라/배포 기록
- 회고 및 실패 기록

이렇게 분류하면 방문자와 검색엔진 모두 콘텐츠 구조를 더 쉽게 이해할 수 있습니다.

### 12. 포트폴리오와 블로그 연결 강화

현재 프로젝트 페이지와 블로그 페이지가 분리되어 있습니다.  
실제 강점은 "프로젝트와 실험 로그가 연결되어 있는 점"이므로, 두 영역을 더 강하게 묶는 편이 좋습니다.

권장 조치:

- 프로젝트 카드에 관련 글 링크 추가
- 블로그 글 하단에 관련 프로젝트 표시
- 특정 프로젝트의 개발 일지를 시리즈로 묶기

## 우선순위 정리

### 즉시 진행 권장

1. `data/siteMetadata.js` 실제 정보 반영
2. `data/authors/default.mdx` 작성자 정보 수정
3. `data/projectsData.ts` 예시 프로젝트 제거
4. `components/Footer.tsx` 원본 템플릿 링크 제거
5. `README.md` 프로젝트 전용 문서로 교체
6. `app/layout.tsx` 폰트 로딩 방식 안정화

### 다음 단계 권장

1. 홈 화면 소개 문구 개편
2. 블로그 포스트 템플릿 개선
3. 프로젝트와 포스트 상호 연결
4. 카테고리/시리즈 구조 강화

## 검토 결과 요약

이 프로젝트는 기반 기술 선택과 구조 자체는 충분히 좋습니다.  
문제는 아키텍처보다 "정체성 정리"와 "템플릿 흔적 제거"에 가깝습니다.

즉, 지금 가장 중요한 것은 새 기능 추가보다 아래 3가지입니다.

1. 실제 개인 사이트 정보로 교체
2. 예시 데이터 제거
3. 콘텐츠와 브랜딩을 본인 방향에 맞게 재정렬

이 세 가지만 먼저 정리해도 프로젝트 완성도는 크게 올라갑니다.
