## 오늘의 핵심 트릭 — sed로 1600줄 포팅

Binance 봇의 `FuturesTrader` 클래스는 1600줄이 넘는다. 이걸 Hyperliquid 버전으로 새로 쓰는 건 하루로 안 끝난다. 그런데 실제로 바뀌어야 하는 건 **client 타입 + 클래스 이름 + WS import**뿐이다. 전략, 필터, 공격 모드, 리스크 관리 — **전부 거래소 독립적**이다.

그래서 이렇게 했다:

```bash
cp core/futures_trader.py core/hl_trader.py

sed -i '' 's/from core\.futures_client import BinanceFuturesClient/from core.hl_client import HyperliquidClient/g' hl_trader.py
sed -i '' 's/^from core\.ws_manager import UserDataWebSocket$/# TODO: HL WS\nUserDataWebSocket = None/' hl_trader.py
sed -i '' 's/^from core\.ws_market import MarketDataWebSocket$/MarketDataWebSocket = None/' hl_trader.py
sed -i '' 's/client: BinanceFuturesClient/client: HyperliquidClient/g' hl_trader.py
sed -i '' 's/class FuturesTrader:/class HyperliquidTrader:/g' hl_trader.py
```

**5초 만에 포팅 완료.** 전략 로직은 한 줄도 안 건드렸다. 이게 가능한 이유는 **client 추상화가 잘 돼 있었기 때문**. `self.client.open_long(symbol, quantity)` 같은 호출은 구현체가 Binance든 Hyperliquid든 동일한 시그니처면 된다.

교훈: 거래소 API 래퍼를 일찍부터 깔끔히 추상화해두면, **거래소 교체는 "설정 변경"에 가까워진다.**

## Hyperliquid와 Binance의 기술적 차이 — 실전 정리

### 1. 인증이 private key

```python
# Binance: 두 개의 문자열
client = Client(api_key, secret_key)

# Hyperliquid: EVM wallet
from eth_account import Account
wallet = Account.from_key(HL_PRIVATE_KEY)
exchange = Exchange(wallet, MAINNET_API_URL)
```

모든 주문은 wallet의 ECDSA 서명으로 검증된다. 다른 말로: **private key = 전부**. 유출되면 끝. 그래서:

- `.env`에 넣고 `chmod 600`
- **API wallet**(서명만 담당) 따로 만들고, **master wallet**(실제 자산)은 건드리지 않기
- `HL_ACCOUNT_ADDRESS`는 master, `HL_PRIVATE_KEY`는 API wallet

### 2. 시장가 주문이 없다

Hyperliquid에는 `MARKET` 타입이 없다. 대신 **IoC(Immediate-or-Cancel) limit order**로 현재가 ±5% 슬리피지 허용해서 즉시 체결:

```python
mark = info.all_mids()["BTC"]
exchange.order(
    name="BTC", is_buy=True, sz=0.01,
    limit_px=mark * 1.05,   # 5% 슬리피지
    order_type={"limit": {"tif": "Ioc"}},
    reduce_only=False,
)
```

### 3. SL/TP는 trigger order

어제 Binance에서 고생한 `closePosition + workingType=MARK_PRICE + priceProtect` 조합의 `-4120` 에러? Hyperliquid에서는 아예 다른 방식이라 해당 없음:

```python
exchange.order(
    name="BTC", is_buy=False, sz=0.01,
    limit_px=49000 * 0.95,
    order_type={
        "trigger": {
            "triggerPx": 49000,
            "isMarket": True,
            "tpsl": "sl",   # 또는 "tp"
        }
    },
    reduce_only=True,
)
```

훨씬 명시적. tpsl 필드 하나로 SL/TP 구분.

### 4. Funding 주기가 1시간

Binance는 8시간마다, Hyperliquid는 **매 시간 정각** 정산. 그래서 기존 `time_filter.py`의 "UTC 00/08/16 ±15분 차단"은 **매 시간 반복**으로 재조정 필요. 이건 Phase 4에서 처리 예정.

### 5. Long/Short Ratio 없음

Binance `globalLongShortAccountRatio` 같은 지표가 Hyperliquid에는 없다. 대신 **Open Interest (OI) 급변**을 대체 시그널로 쓸 예정. 스퀴즈 감지 로직은 비슷한 효과.

## 배운 점

### 1. 추상화는 **미래의 나**를 위한 것

Binance 클라이언트를 객체로 추상화해두지 않았다면 오늘 포팅은 며칠 걸렸을 거다. 몇 달 전에 클라이언트를 분리해둔 과거의 내가 오늘 나를 살렸다. **모든 외부 서비스는 얇은 래퍼로 감싸라.**

### 2. sed는 강력하지만 책임은 커진다

5초 포팅은 멋지지만, 치환된 부분을 **한 번은 직접 읽어야 한다**. 오늘은 grep으로 주요 심볼만 확인했는데, 세부 검증은 실제 가동 + import 테스트로 한다. 게으름의 대가는 버그다.

### 3. 솔로 루프(MODE B)도 충분히 강력하다

원래 `/infinite-feedback`는 tmux 멀티 에이전트를 주로 쓰지만, macOS + non-tmux 환경에서도 MODE B로 충분히 체계적으로 작업할 수 있었다. **멀티 에이전트가 항상 정답은 아니다.** 잘 분할된 단일 스레드가 더 예측 가능하다.

### 4. 블로그가 미래의 로드맵이 된다

어제 Part 4에서 "Hyperliquid 포팅 추천"이라고 써놓고, 오늘 실제로 했다. **글로 선언한 것의 실행률이 안 쓴 것보다 훨씬 높다.** 이게 오늘 작업의 동기 절반 이상이었다.
