## 문제 상황

`supabase db push` 실행 중 다음 에러 발생:

```
NOTICE (42P07): relation "housing_listings" already exists, skipping
ERROR: column "is_visible" does not exist (SQLSTATE 42703)
At statement: create index if not exists idx_housing_visible
  on public.housing_listings(is_visible, region, created_at desc)
```

마이그레이션이 중간에 실패해서 `supabase_migrations` 히스토리에 기록되지 않은 상태였고,
재실행 시 `CREATE TABLE IF NOT EXISTS`는 통과하지만 이미 존재하는 테이블에 `is_visible` 컬럼이 없어서 인덱스 생성이 실패했다.

## 원인 분석

1. 이전 마이그레이션이 도중에 실패 → DB에 부분 적용된 상태
2. `CREATE TABLE IF NOT EXISTS`는 기존 테이블을 그냥 통과 (컬럼 추가 없이)
3. 이후 `CREATE INDEX`가 존재하지 않는 컬럼을 참조 → 에러

Supabase의 마이그레이션은 **트랜잭션 단위로 롤백되지 않는 DDL**이 포함되어 있어,
실패 시 일부만 적용된 채로 남는 경우가 있다.

## 해결 방법

### 1. 메인 마이그레이션을 idempotent하게 수정

컬럼 추가 부분을 `DO $$` 블록으로 감싸서 이미 존재하면 스킵하도록 변경:

```sql
do $$
begin
  if not exists (
    select 1 from information_schema.columns
    where table_schema = 'public'
      and table_name = 'housing_listings'
      and column_name = 'is_visible'
  ) then
    alter table public.housing_listings add column is_visible boolean default true;
  end if;
end $$;
```

### 2. 복구 마이그레이션 별도 생성

`20260408000001_ink_repair_housing.sql` 파일로 분리해서 안전하게 추가:

```sql
-- CREATE POLICY도 중복 방지
do $$
begin
  if not exists (
    select 1 from pg_policies
    where schemaname = 'public'
      and tablename = 'housing_listings'
      and policyname = '공개 숙소 읽기'
  ) then
    create policy "공개 숙소 읽기" on public.housing_listings
      for select using (is_visible = true);
  end if;
end $$;
```

## 배운 점

- Supabase 마이그레이션은 실패해도 부분 적용될 수 있다 → **처음부터 idempotent하게 작성**
- `CREATE TABLE IF NOT EXISTS`만으로는 부족 — 컬럼 추가도 `DO $$` 블록으로 감싸야 안전
- `CREATE POLICY`도 중복 실행 시 에러 → `pg_policies` 테이블 조회로 존재 여부 확인 필요
- 복잡한 마이그레이션은 별도 repair 마이그레이션 파일을 만드는 게 깔끔하다
