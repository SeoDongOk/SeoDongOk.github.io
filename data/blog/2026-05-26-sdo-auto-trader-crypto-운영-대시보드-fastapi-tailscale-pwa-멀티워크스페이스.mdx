## 문제 상황
기존 `sdo_auto_trader_crypto`는 라즈베리파이에서 자동매매 봇이 잘 돌고 있었지만, 운영 관점에서는 불편한 점이 많았다.

- 상태 확인을 위해 매번 `ssh home`으로 접속해야 했다.
- 현재 포지션, 최근 거래, 에러 로그를 한 화면에서 보기 어려웠다.
- 수동 제어(재시작, 추가 진입, 부분 청산, 전체 청산)가 없었다.
- 외부에서 접속하려면 포트포워딩 같은 네트워크 이슈를 먼저 해결해야 했다.
- 나중에 ELW, KOS 같은 다른 작업영역까지 확장할 구조도 필요했다.

즉, 자동매매 전략 자체보다도 **운영 플랫폼**이 필요한 시점이었다.

## 원인 분석
문제의 핵심은 봇이 잘못된 게 아니라, **운영 인터페이스가 없었다는 점**이었다.

기존 구조는 아래에 가까웠다.

- Python 봇 실행
- 로그 파일 저장
- SQLite 거래 기록 저장
- 사용자는 SSH로 접속해서 직접 확인

이 구조는 초반 실험에는 충분하지만, 운영 기능이 늘어날수록 다음 문제가 생긴다.

1. 상태 확인이 느리다.
2. 로그/포지션/수익률 확인이 분산된다.
3. 실수로 잘못된 명령을 치기 쉽다.
4. 모바일에서 확인하기 어렵다.
5. 다른 시장(ELW, KOS)으로 확장하기 어렵다.

## 해결 방법
이번 작업에서는 단순 API 추가가 아니라, **운영용 관리자 플랫폼**을 단계적으로 붙였다.

### 1. FastAPI 기반 운영 API 구축
먼저 `api/` 폴더를 만들고 FastAPI 기반 엔드포인트를 구성했다.

주요 API:
- `/api/health`
- `/api/bot/status`
- `/api/bot/start`
- `/api/bot/stop`
- `/api/bot/restart`
- `/api/logs/recent`
- `/api/logs/errors`
- `/api/positions`
- `/api/pnl/today`
- `/api/trades/recent`
- `/api/manual/*`

그리고 Basic Auth를 붙여 내부 API를 보호했다.

### 2. 로그인 가능한 웹 관리자 대시보드 구성
FastAPI 템플릿 기반으로 `/login`, `/dashboard`를 구성했다.

대시보드에서 볼 수 있는 항목:
- 봇 PID / 봇 상태
- 열린 포지션 수
- 오늘 손익
- 평균 수익률 / 승률
- 누적 / 주간 / 월간 수익
- 심볼별 수익률
- 최근 거래 내역
- 최근 Trader / Error 로그

또한 아래 수동 액션도 추가했다.
- 추가 진입
- 부분 청산
- 전체 청산
- 신규 진입 / 올인
- 기본 비율 변경

### 3. 수동 액션 보호장치 추가
실거래 리스크가 있기 때문에 UI에서 바로 주문이 나가는 구조는 위험하다.

그래서 두 단계 보호를 넣었다.

- 액션 실행 전 브라우저 `confirm()`
- `prod` 환경에서는 확인 문구 입력 강제

서버 쪽에서도 확인 문구를 다시 검사하게 해서, 프론트 우회 호출도 막는 방향으로 정리했다.

### 4. Tailscale로 외부 안전 접속 구성
포트포워딩 없이 외부에서 접근할 수 있도록 Tailscale을 사용했다.

구성:
- 맥북에 Tailscale 설치
- 라즈베리파이에 Tailscale 설치
- 같은 Tailnet에 연결
- `http://<tailscale-ip>/login` 형태로 접근

이 방식의 장점은 다음과 같다.
- 공유기 포트포워딩 불필요
- 외부 노출 위험 감소
- 모바일에서도 접속 가능

### 5. 모바일 대응 + PWA 적용
운영 도구는 결국 휴대폰에서도 봐야 한다.

그래서 다음을 적용했다.
- 반응형 레이아웃
- 모바일 포지션 카드 UI
- 하단 고정 네비게이션
- 30초 자동 갱신
- 홈 화면에 추가 가능한 PWA 설정

적용 파일:
- `manifest.webmanifest`
- `service worker`
- `icon.svg`

### 6. 멀티 워크스페이스 구조 도입
처음에는 crypto 전용이었지만, 이후 ELW/KOS까지 확장하려면 공통 운영 레이어가 필요했다.

그래서 워크스페이스 개념을 도입했다.

- `crypto`
- `elw`
- `kos`

현재 상태:
- `crypto`는 실제 구현체 연결
- `elw`, `kos`는 placeholder 연결
- 대시보드 상단 selector로 작업영역 전환 가능

이 구조의 장점은 UI는 유지하면서도, 시장별 서비스 구현만 갈아끼울 수 있다는 점이다.

## 배운 점
### 1. 운영 플랫폼은 전략만큼 중요하다
자동매매는 진입/청산 로직도 중요하지만, 실제 운영에서는
- 상태 확인
- 로그 추적
- 수동 개입
- 수익률 분석
이 더 자주 필요했다.

### 2. 포트포워딩보다 Tailscale이 훨씬 현실적이었다
개발/운영 초기에는 퍼블릭 오픈보다 Tailscale이 훨씬 안전하고 빠르다.
특히 개인 운영 환경에서는 체감 차이가 크다.

### 3. 수동 액션은 반드시 보호장치가 있어야 한다
추가 진입, 전체 청산, 올인 같은 기능은 편하지만 위험하다.
따라서 확인 모달과 `prod` 보호 문구는 필수에 가깝다.

### 4. 처음부터 멀티 워크스페이스를 염두에 두는 게 좋다
지금은 crypto만 연결돼 있어도,
초기에 `workspace` 추상화를 해두면 나중에 ELW, KOS를 붙일 때 대규모 재작업을 줄일 수 있다.

## 현재 남은 작업
- `/api/workspaces` 메타 API 추가
- ELW 실제 서비스 구현체 연결
- KOS 실제 서비스 구현체 연결
- 일별 수익 곡선 차트 추가
- 심볼별 필터/정렬
- 액션 로그 DB 저장

## 마무리
이번 작업으로 `sdo_auto_trader_crypto`는 단순한 라즈베리파이 봇에서,
**웹 기반 운영 플랫폼을 가진 자동매매 시스템**으로 한 단계 올라왔다.

아직 ELW/KOS는 placeholder 수준이지만, 구조적으로는 이미 확장 가능한 형태가 됐다.
다음 단계는 멀티 워크스페이스를 실제로 연결하고, 수익률 시각화까지 강화하는 것이다.
