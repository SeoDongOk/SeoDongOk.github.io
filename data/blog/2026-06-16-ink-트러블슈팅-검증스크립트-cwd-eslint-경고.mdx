## 문제 상황

`user_profiles` privacy hardening 후 SQL 정적 검증, 테스트, ESLint를 한 번에 실행하는 과정에서 SQL 파일 확인 스크립트가 `FileNotFoundError`로 실패했다. 같은 검증 단계에서 `profileApi.ts`에는 `Expected 'undefined' and instead saw 'void'` ESLint 경고도 발생했다.

## 원인 분석

명령의 `workdir`를 `vietnam-in-korea-superapp-rn`으로 설정한 상태에서 Python 스크립트가 다시 `vietnam-in-korea-superapp-rn/supabase/...` 경로를 붙여 파일을 찾았다. 즉 현재 작업 디렉터리 기준과 상대 경로가 중복된 것이 원인이었다.

ESLint 경고는 기존 함수 시그니처 호환을 위해 `params.locale`을 `void params.locale`로 소비한 코드가 `no-void` 규칙에 걸린 것이었다.

## 해결 방법

SQL 검증은 레포 루트에서 실행하거나, RN 앱 디렉터리 기준 상대 경로를 맞추도록 분리했다. 이후 파일 존재, 세미콜론 수, 괄호 균형, 필수 RPC 이름 포함 여부를 정상 확인했다.

`void params.locale`은 제거했다. `params` 객체는 `params.userId` 접근에 이미 사용되고 있어 별도 no-op 소비가 필요하지 않았다.

## 배운 점

검증 명령을 여러 개 체인으로 묶을 때는 각 명령의 CWD 기준을 명확히 해야 한다. 또한 사용하지 않는 호환성 파라미터를 처리할 때 `void` no-op보다 함수 시그니처 유지 + 실제 사용 필드만 접근하는 편이 lint 친화적이다.
