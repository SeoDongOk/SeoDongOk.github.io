[지난 글(0)](/blog/2026-06-21-aws-계정-초기세팅-iam-리전제한-비용알람)에서 IAM 사용자·리전 제한·비용 알람을 끝냈다. 이번엔 본격적으로 **네트워크(VPC)**와 **서버(EC2)**를 만든다. 그리고 이 과정에서 두 번 크게 삽질했다 — **VPC를 엉뚱한 리전에 만든 것**과 **프리티어가 이미 만료돼 있던 것**.

## 1. VPC — 봇 인프라의 네트워크 토대

VPC는 **AWS 안에 만드는 나만의 격리된 가상 네트워크**다. 집의 공유기 네트워크라고 보면 된다. 구성 요소:

- **CIDR**: IP 주소 범위 (`10.0.0.0/16`)
- **서브넷**: 네트워크를 나눈 칸 (퍼블릭 = 인터넷 연결 / 프라이빗 = 격리)
- **인터넷 게이트웨이(IGW)**: 인터넷 출입구
- **라우팅 테이블**: "어디로 가려면 어느 문으로" 길 안내

콘솔의 **`Create VPC` → `VPC and more`** 마법사를 쓰면 서브넷·IGW·라우팅을 한 번에 만들어 준다. 봇용 설계는 단순하게:

```
VPC  10.0.0.0/16
 └ 퍼블릭 서브넷 1개  (+ Elastic IP / 퍼블릭 IP)
 └ IGW + 라우팅(0.0.0.0/0 → IGW)
 └ NAT Gateway: None
 └ AZ: 1 / Private subnet: 0
```

### NAT Gateway는 왜 None?

NAT Gateway는 **프라이빗 서브넷이 인터넷으로 나가는(outbound) 단방향 출구**다. 우리는 봇을 **퍼블릭 서브넷**에 둘 거라 IGW로 직접 나갈 수 있어서 NAT가 필요 없다. 게다가 **NAT는 월 ~$32**라, 봇 한 대 소액 운영엔 과한 비용이다. (보안을 빡세게 가려고 봇을 프라이빗에 숨길 때나 쓴다)

### CIDR /16 vs /24, 뭐가 싸냐?

**둘 다 무료다.** VPC·서브넷·라우팅·IGW는 전부 공짜고, CIDR 크기는 "내부 사설 IP를 몇 개 쓸 수 있나"일 뿐 요금과 무관하다. 그래서 여유 있는 `/16`(65,536개)으로 잡았다. (진짜 비용은 EC2·퍼블릭 IP·NAT·데이터 전송에서 나온다)

## 2. ⚠️ 최대 삽질: VPC를 시드니에 만들었다

VPC를 다 만들고 EC2를 띄우려는데, **Network settings의 VPC 드롭다운에 방금 만든 `trading-vpc`가 안 보였다.** default VPC(`172.31.0.0/16`)만 떴다.

원인은 **리전 불일치**였다. **VPC는 리전마다 따로 존재**하는데, 나는 VPC를 **시드니(ap-southeast-2)**에서 만들고 EC2는 서울에서 만들려 했던 것. (심지어 시드니엔 예전에 만든 `auto-trader` EC2와 보안그룹 6개가 남아 있었다 — 이것도 같이 정리했다. Terminate하니 EBS 볼륨까지 깨끗이 삭제돼서 과금원은 0이 됐다.)

**해결**: 서울로 리전을 바꾸고 `trading-vpc`를 다시 만들었다. VPC 다시 만드는 건 5분이면 끝난다.

> **교훈**: 모든 리소스를 **같은 리전(서울)**에. 작업 전 **우측 상단 리전을 항상 확인**하는 습관이 필요하다. "리전마다 리소스가 따로"라는 걸 몸으로 배웠다.

## 3. EC2 — 서버 만들기

`Launch instance`로 들어가서 AMI·인스턴스 타입·키페어·보안그룹을 정한다.

### ⚠️ 삽질: AMI 검색하니 "with SQL" x86만 나옴

검색창에 `22.04`/`24.04`를 치면 **Marketplace 변형(`with SQL Server`, x86)**이 나온다. 순정 Ubuntu는 검색이 아니라 **Quick Start 탭의 `Ubuntu`**에서 골라야 한다. 거기서 **`Ubuntu Server 24.04 LTS` 순정 + Architecture 선택**.

### 보안그룹: SSH를 0.0.0.0/0으로 열면 안 된다

기본값이 SSH `Anywhere(0.0.0.0/0)`였다. 이러면 **전 세계 봇들이 SSH 무차별 공격**을 시도한다. **Source type을 `My IP`**로 바꿔서 내 집 IP에서만 접속 가능하게 했다. (집 IP가 바뀌면 보안그룹에서 IP만 갱신하면 된다)

## 4. ⚠️ 삽질: 프리티어가 이미 만료돼 있었다

인스턴스 타입을 `t3.micro`(프리티어 무료)로 고르려다 "어, 나 프리티어 아닌데?" 싶었다. **Billing → Free Tier** 페이지를 보니 KMS·Glue·SQS 같은 **"always free"(영구 무료)** 항목만 있고, **EC2/EBS의 "12개월 프리티어"는 목록에 없었다** → 가입 12개월이 지나 **만료된 것**이었다.

그러면 비용 계산이 달라진다 (서울, 24시간 가동 기준):

| 인스턴스            | 인스턴스(24h) | +EBS 8GB +퍼블릭 IP | 합계          |
| ------------------- | ------------- | ------------------- | ------------- |
| t3.micro (x86)      | ~$9.4         | +$4.2               | **~$13.6** ❌ |
| **t4g.micro (ARM)** | ~$6.0         | +$4.2               | **~$10.2** ✅ |

목표는 **월 $10 이하**라, **t4g.micro(ARM Graviton)**으로 바꿨다. 핵심은:

- **성능 차이는 봇 용도엔 사실상 없다.** 둘 다 2vCPU/1GB 동일하고, t4g가 가성비 20~40% 좋으면서 실성능도 동급 이상. 봇은 Python·네트워크 I/O 위주라 체감 차이 0.
- **봇이 원래 라즈베리파이(ARM)에서 돌던 거라**, ARM(t4g)이 오히려 **기존 환경과 동일 아키텍처** → 호환성 걱정이 없다. (numpy·pandas 등 ARM wheel 다 있음)
- 더 싸고(월 ~$3↓) 성능 동급 + 같은 아키텍처. 안 쓸 이유가 없다.

대신 t4g는 ARM이라 **AMI를 Ubuntu 24.04 `64-bit (Arm)`로 다시 골라야** 했다 (x86 AMI면 "아키텍처 미지원" 에러가 난다).

### EBS도 프리티어 기준이 달라진다

프리티어면 EBS 30GB까지 무료라 넉넉히 잡으려 했지만, **만료라 8GB로** 줄였다 (30GB는 과금). 8GB gp3 ≈ $0.6/월.

## 5. 결과

서울에 **`trading-server`(t4g.micro, ARM, Ubuntu 24.04)**가 `Running`으로 떴고, 퍼블릭 IP도 받았다. 월 비용은 ~$10 선.

## 배운 점

- **리전 통일이 1번이다.** VPC도 EC2도 같은 리전에 있어야 서로 연결된다. 우측 상단 리전 확인을 습관화.
- **프리티어 만료를 먼저 확인하자.** Free Tier 페이지에 "always free"만 보이면 12개월 무료는 끝난 것. 비용 설계 전제가 바뀐다.
- **ARM(Graviton)이 싸고 성능 동급**이다. 특히 봇이 라즈베리파이(ARM) 출신이면 ARM이 오히려 자연스럽다. 막연히 x86 고를 이유가 없었다.
- **비용은 인스턴스·퍼블릭 IP·NAT에서 샌다.** NAT 회피, 퍼블릭 IP 1개, EBS 최소, ARM 인스턴스로 $10 선을 맞췄다.
- 삽질의 공통점은 또 **"리전·아키텍처·프리티어" 같은 전제를 안 챙긴 것**이었다. 만들기 전에 전제부터 확인하면 절반은 안 틀린다.
