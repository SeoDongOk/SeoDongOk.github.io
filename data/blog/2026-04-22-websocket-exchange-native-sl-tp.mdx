> 이 포스트는 [팀 리뷰로 바꾼 자동매매 봇 — 온체인 데이터를 필터로 쓰는 3가지 방법](./2026-04-22-onchain-filter-team-review)의 **Part 2**다. Part 1에서는 "진입을 어떻게 고를 것인가"를 다뤘고, 여기서는 "진입한 뒤에 죽지 않고 살아남을 것인가"를 다룬다.

## 문제 정의 — 팀 리뷰에서 나온 가장 무서운 한 줄

백엔드 엔지니어 강민수의 지적:

> "SL/TP를 거래소에 주문으로 안 걸고 1분 폴링만 하면, 봇 다운 시 청산 리스크 그대로 노출됩니다."

리스크 매니저 윤재호도 같은 방향:

> "20x 레버리지 + ISOLATED = 가격 -5% 역방향 시 증거금 100% 증발. 그런데 방어선이 1분마다 돌아가는 파이썬 프로세스 하나뿐이에요. 네트워크 단절 2분 → SL 뚫리고도 청산 안 됨 → 청산가까지 방치."

### 기존 구조가 왜 위험한가

```
┌─────────────────────────────────────────────┐
│ APScheduler (매 1분)                        │
│   └─ run_cycle()                           │
│        ├─ 현재가 조회                       │
│        ├─ if price <= stop_loss:           │
│        │     close_position("손절")         │  ← 이게 유일한 방어선
│        └─ ...                              │
└─────────────────────────────────────────────┘
```

**장애 시나리오**:

1. BTCUSDT 진입 중, 봇 프로세스가 OOM으로 kill됨
2. 가격이 -2% 역방향 급락 (15분봉에서 충분히 나올 수 있음)
3. 20x 레버리지에서 -2% 가격 = 마진 -40% 손실 = SL(-15%) 이미 관통
4. 하지만 **봇이 죽어있어서 청산 안 됨**
5. 가격이 -5% 도달 → 거래소가 자동 liquidation → **마진 100% 증발**

봇 크래시, systemd 재시작 사이 공백 10초, 라즈베리파이 정전, 심지어 Claude Code로 `bash stop.sh → start.sh`하는 그 사이 5초 — **전부 위험 구간**이다.

## 해결책 — 2중화

두 개의 독립적인 안전장치를 만들었다:

1. **거래소에 SL/TP를 실제 주문으로 걸어둠** — 봇이 죽어도 거래소가 알아서 청산
2. **WebSocket User Data Stream** — 수동매매 / 외부 청산을 실시간 감지

### Layer 1: 거래소 STOP_MARKET + TAKE_PROFIT_MARKET

기존 코드에 `place_stop_loss`, `place_take_profit` 메서드가 **이미 있었다**. 왜 안 쓰고 있었냐? 주석 하나에서 이유를 찾았다:

```python
# SL/TP: Binance가 STOP_MARKET/TAKE_PROFIT_MARKET을 Algo Order API로만 허용(-4120)하므로
# 거래소 주문 없이 1분마다 내부 모니터링(_check_sl_tp)으로만 손절/익절 처리
```

`-4120` 에러. 과거에 시도했다가 에러 나서 비활성화된 상태. 하지만 실제로 Binance Futures는 STOP_MARKET을 **지원한다**. 단 몇 가지 파라미터를 제대로 써줘야 한다.

**정답 조합**:

```python
def place_stop_loss(self, symbol, side, quantity, stop_price) -> Optional[dict]:
    close_side = "SELL" if side == "LONG" else "BUY"
    precision = self.get_price_precision(symbol)
    formatted_price = f"{stop_price:.{precision}f}"
    try:
        order = self._client.futures_create_order(
            symbol=symbol,
            side=close_side,
            type="STOP_MARKET",
            stopPrice=formatted_price,
            closePosition="true",      # quantity/reduceOnly 대체
            workingType="MARK_PRICE",   # Liquidation 가격과 일치
            priceProtect="true",        # 스파이크 방지
        )
        return order
    except BinanceAPIException as e:
        logger.error(f"거래소 손절 주문 실패 ({symbol}): {e}")
        return None   # 치명적이지 않음 — 내부 폴링이 fallback
```

세 가지 포인트:

1. **`closePosition="true"`** → 수량 지정 불필요, 가격 도달 시 전체 포지션 청산
2. **`workingType="MARK_PRICE"`** → Liquidation과 같은 기준가를 씀. CONTRACT_PRICE(체결가)는 조작 가능해서 위험
3. **`priceProtect="true"`** → 비정상적 스파이크로 트리거되는 걸 방지

그리고 **실패해도 None을 반환해서 포지션 오픈 자체는 계속 진행**되도록. 내부 1분 폴링(`_check_sl_tp`)이 여전히 fallback으로 작동한다. 이게 진짜 2중화다 — 한쪽이 실패해도 다른 쪽이 커버.

### Layer 2: WebSocket User Data Stream

봇이 혼자 매매하면 Layer 1로 충분한데, **내가 수동으로 바이낸스에서 매매**하면 어떻게 될까?

- 봇은 내 수동 포지션을 모름
- 그 포지션에 SL/TP가 안 걸림
- 봇이 중복 진입 시도할 수 있음

기존 `_sync_positions_from_exchange()` 메서드에 외부 진입 감지 로직이 있지만 **봇 시작 시 1회만** 실행된다. 실시간은 아님.

해결: **Binance Futures User Data Stream** 구독. Private WebSocket으로 계정 이벤트(주문 체결, 포지션 변화, 마진 콜)를 푸시받는다.

#### WS Manager (core/ws_manager.py)

```python
from binance import ThreadedWebsocketManager

class UserDataWebSocket:
    def __init__(self, api_key, secret_key, event_queue, *, testnet=False, max_reconnect=5):
        self._api_key = api_key
        self._secret_key = secret_key
        self._queue = event_queue  # 스레드 세이프 Queue
        self._twm: Optional[ThreadedWebsocketManager] = None
        self._running = False
        ...

    def _supervise(self):
        """연결 끊김 감지 + 재연결 루프"""
        while self._running and self._reconnect_count <= self._max_reconnect:
            try:
                self._connect()
                while self._running:
                    if self._twm is None or not self._twm.is_alive():
                        break
                    time.sleep(5)
            except Exception as e:
                logger.error(f"WS supervisor 예외: {e}", exc_info=True)
            # exponential backoff
            self._reconnect_count += 1
            wait_sec = min(30, 5 * self._reconnect_count)
            time.sleep(wait_sec)

    def _on_message(self, msg):
        """WS 스레드 → 메인 스레드로 이벤트 전달"""
        try:
            event_type = msg.get("e")
            if event_type == "ACCOUNT_UPDATE":
                self._queue.put({"type": "ACCOUNT_UPDATE", "data": msg})
            elif event_type == "ORDER_TRADE_UPDATE":
                self._queue.put({"type": "ORDER_TRADE_UPDATE", "data": msg})
            elif event_type == "listenKeyExpired":
                self._twm.stop()  # supervisor가 재연결 처리
            elif event_type == "MARGIN_CALL":
                logger.error(f"⚠️ MARGIN_CALL: {msg}")
                self._queue.put({"type": "MARGIN_CALL", "data": msg})
        except Exception as e:
            # 콜백에서 예외 나면 WS가 죽으니 반드시 잡기
            logger.error(f"WS 콜백 예외: {e}", exc_info=True)
```

#### Event Queue 패턴 — 왜?

WS 콜백은 **별도 스레드**에서 실행된다. `self._positions` 딕셔너리를 여기서 직접 수정하면 메인 스레드와 race condition 생긴다. 그래서:

- WS 스레드: `queue.put()`만 호출 (thread-safe)
- 메인 스레드 (APScheduler job): `run_cycle` 시작 시 `_drain_ws_events()`로 큐 소진

```python
def run_cycle(self, symbol: str) -> None:
    # WS 이벤트 먼저 처리: 수동 진입/외부 청산 반영
    self._drain_ws_events()
    ...

def _drain_ws_events(self) -> None:
    while True:
        try:
            event = self._ws_events.get_nowait()
        except Empty:
            return
        try:
            etype = event.get("type")
            if etype == "ACCOUNT_UPDATE":
                self._handle_account_update(event["data"])
            elif etype == "ORDER_TRADE_UPDATE":
                self._handle_order_update(event["data"])
        except Exception as e:
            logger.error(f"WS 이벤트 처리 예외: {e}", exc_info=True)
```

Lock 없이 동시성 문제 해결. 디버깅 쉽고, 콜백에서 뭔가 터져도 WS 연결은 살아있다.

#### 수동매매 감지 핵심 로직

`ACCOUNT_UPDATE` 이벤트의 `msg['a']['P']`에 포지션 배열이 들어온다:

```python
def _handle_account_update(self, msg: dict) -> None:
    positions = msg.get("a", {}).get("P", [])
    for p in positions:
        symbol = p.get("s")
        if symbol not in self.settings.symbols:
            continue
        amt = float(p.get("pa", 0))  # positionAmt
        entry = float(p.get("ep", 0))
        in_bot = symbol in self._positions

        if amt == 0 and in_bot:
            # 외부 청산 감지 (수동 청산, 거래소 청산, SL/TP 주문 체결)
            db_pos = self._positions.pop(symbol)
            # PnL 계산 + 기록
            exit_price = self._last_prices.get(symbol, db_pos.entry_price)
            pnl = db_pos.unrealized_pnl(exit_price)
            self._recorder.record_trade(...)
            logger.warning(f"🔔 외부 청산 감지 (WS): {symbol} {db_pos.side} @ {exit_price}")

        elif amt != 0 and not in_bot:
            # 수동 진입 감지!
            side = "LONG" if amt > 0 else "SHORT"
            qty = abs(amt)
            sl = self._calc_sl(entry, side)
            tp = self._calc_tp(entry, side)

            self._positions[symbol] = FuturesPosition(
                symbol=symbol, side=side, entry_price=entry,
                quantity=qty, stop_loss=sl, take_profit=tp,
                entry_tag="manual_ws_detected", peak_price=entry,
            )
            # 거래소 SL/TP 주문도 바로 걸기
            self.client.place_stop_loss(symbol, side, qty, sl)
            self.client.place_take_profit(symbol, side, qty, tp)

            logger.warning(f"🔔 수동 진입 감지 (WS): {symbol} {side} {qty}개 @ {entry}")
```

**"내가 바이낸스 앱에서 매매해도, 봇이 1초 안에 감지해서 SL/TP를 자동으로 걸어준다."**

## 배운 점

### 1. 리뷰 한 번이 3개월 개발보다 낫다

`place_stop_loss`, `place_take_profit` 메서드는 **이미 코드에 있었다**. 과거의 내가 `-4120` 에러 보고 "안 되네" 하고 주석 처리했던 거다. 강민수가 "이거 왜 비활성화된 거예요?"라고 짚어주지 않았으면 앞으로도 계속 방치됐을 거다.

팀 리뷰의 가치는 **새 코드를 쓰게 만드는 것**이 아니라 **죽어있던 코드를 깨우게 만드는 것**이기도 하다.

### 2. "실패 = 관대"의 원칙

거래소 SL/TP 주문은 실패 시 `None`을 반환한다. 예외를 던지지 않는다. 이유:

- Binance API는 rate limit, 네트워크 flap, 간헐적 5xx가 있다
- SL/TP 주문이 실패했다고 포지션 오픈까지 롤백하면 "진입 자체가 실패"로 변질
- 대신 내부 폴링이 fallback으로 작동 — **2중화의 진짜 의미는 한쪽 실패 시 다른 쪽이 커버**

```python
# _open_position에서:
if not self.settings.is_paper_trading() and self.settings.exchange_sl_tp_enabled:
    try:
        self.client.place_stop_loss(symbol, side, quantity, sl)
        self.client.place_take_profit(symbol, side, quantity, tp)
    except Exception as e:
        logger.error(f"{symbol} 거래소 SL/TP 실패 (내부 폴링으로 fallback): {e}")
```

### 3. Event Queue는 동시성 문제의 정답

WS 콜백에서 직접 `_positions`를 수정하고 싶은 유혹이 강했다. 코드가 간단해 보이니까. 근데:

- 콜백에서 예외 나면 WS 스레드 자체가 죽음
- 메인 스레드가 `_positions[symbol]` 접근하는 동안 다른 스레드가 `pop` 하면 race
- Lock을 걸기 시작하면 모든 메서드에 퍼져나감

Queue 하나 끼워넣는 대신 생각의 부하가 절반으로 줄어든다. 비동기 시스템에서는 **"언제 처리하는가"를 명시적으로 분리**하는 게 대부분의 경우 더 싸다.

### 4. Binance `-4120`의 진실

과거에 본 `-4120` 에러의 실제 원인은 **필수 파라미터 누락**이었다. 특히 `workingType`이 default로 `CONTRACT_PRICE`로 들어가는데, 이게 일부 검증에서 이상치로 판정되어 reject 되는 케이스가 있다. `MARK_PRICE`로 명시하고 `priceProtect=true` 붙이니까 바로 성공.

**Binance API 에러 메시지는 대체로 정확하지 않다.** `-4120`은 "Order type/side invalid"라고 뜨는데 실제는 "필수 파라미터 조합 이상". 한국어 블로그 검색보다 [공식 GitHub issue + 영어 Stack Overflow](https://github.com/binance/binance-spot-api-docs/issues)에서 최신 사례를 찾는 게 빠르다.

- **강민수 미반영**: `except Exception`의 광범위한 catch를 구체 예외로 좁히기
- **박현석 미반영**: Prometheus 메트릭 + Grafana 대시보드 (설계는 `docs/grafana-integration.md`에 완성, 실제 설치는 다음 작업 사이클)

그리고 오늘의 가장 큰 숙제: **과거 커밋에 노출된 API Key 회전**. 이건 코드가 아니라 사람 손으로 해야 하는 일이다. Binance API Management → Delete → 새 Key 생성.

## 최종 구조

```
┌──────────────────────────────────────────────────────────────┐
│ Layer 0: 봇 내부 SL/TP 폴링 (1분 간격, 기존)                  │
│  - network 문제 시 last resort                               │
├──────────────────────────────────────────────────────────────┤
│ Layer 1: 거래소 STOP_MARKET + TAKE_PROFIT_MARKET (신규)       │
│  - closePosition + MARK_PRICE                                │
│  - 봇이 죽어도 거래소가 자동 청산                             │
├──────────────────────────────────────────────────────────────┤
│ Layer 2: WebSocket User Data Stream (신규)                   │
│  - ACCOUNT_UPDATE / ORDER_TRADE_UPDATE                       │
│  - 수동매매 / 외부청산 실시간 감지                            │
│  - Event Queue 기반 스레드 세이프                             │
└──────────────────────────────────────────────────────────────┘
```

3중으로 안전장치를 두는 이유는 하나다. **20x 레버리지에서 한 번의 실수는 복구 불가**하니까.

Part 1에서 "진입을 잘 고르는 법"을 얘기했고, Part 2에서는 "진입 후에 죽지 않는 법"을 얘기했다. 내일은 **"죽을 뻔한 걸 어떻게 감지하는가"** — Prometheus + Grafana로 모니터링 레이어를 올리는 얘기가 될 예정이다.
