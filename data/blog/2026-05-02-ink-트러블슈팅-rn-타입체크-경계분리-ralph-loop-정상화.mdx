## 문제 상황

INK React Native 프로젝트에 Ralph Loop를 붙인 뒤 `npm test`는 통과했지만, `lint`와 `tsc`가 앱 기능과 직접 관계없는 영역까지 함께 검사하면서 반복 루프가 계속 막혔다.

대표적으로 두 부류가 섞여 있었다.

- `supabase/functions` 아래 Deno Edge Functions
- `app/` 아래 `expo-router` 실험 파일

RN 앱 자체는 `src/`와 `App.tsx`를 중심으로 돌아가는데, 검증 단계가 저장소 전체를 한 번에 훑으면서 아래 같은 에러가 발생했다.

```text
Cannot find module 'expo-router'
Cannot find module 'https://esm.sh/@supabase/supabase-js@2'
Cannot find name 'Deno'
```

이 상태에서는 AI가 앱 기능을 잘 고쳐도 Ralph Loop가 계속 실패 로그만 다시 먹는 비효율적인 루프가 됐다.

## 원인 분석

원인은 검증 기준이 실제 실행 단위와 맞지 않았기 때문이다.

- `tsconfig.json`이 `**/*.ts`, `**/*.tsx`를 전부 포함하고 있었다.
- `eslint .`도 Deno 함수와 RN 앱을 같은 규칙셋으로 검사하고 있었다.
- `react-native-iap`, `@react-native-seoul/kakao-login`은 런타임 의존은 있지만 타입 선언이 정리되지 않아 앱 타입체크까지 흔들고 있었다.

즉, 앱 코드의 품질 문제가 아니라 **검증 경계 설계가 너무 넓었던 것**이 핵심이었다.

## 해결 방법

### 1. RN 앱 전용 타입체크 설정 분리

`tsconfig.app.json`을 새로 만들어 RN 앱에서 실제로 검증해야 하는 범위만 포함했다.

```json
{
  "extends": "./tsconfig.json",
  "include": [
    "App.tsx",
    "react-native-config.d.ts",
    "src/**/*.ts",
    "src/**/*.tsx",
    "__tests__/**/*.ts",
    "__tests__/**/*.tsx"
  ],
  "exclude": ["app", "supabase/functions", "android", "ios", "node_modules"]
}
```

그리고 `package.json`에 전용 스크립트를 추가했다.

```json
{
  "scripts": {
    "typecheck": "tsc --noEmit --project tsconfig.app.json"
  }
}
```

### 2. ESLint에서 RN 루프와 무관한 경로 제외

`.eslintignore`에 아래를 추가했다.

```text
supabase/functions/**
.bkit/**
```

이렇게 하면 React Native용 ESLint가 Deno Edge Functions까지 검사하면서 생기는 불필요한 에러를 막을 수 있다.

### 3. 누락된 네이티브 모듈 타입 선언 보강

`src/types/native-modules.d.ts`를 추가해 `react-native-iap`, `@react-native-seoul/kakao-login` 선언을 보강했다.

이 작업으로 패키지가 실제 네이티브 환경에 있더라도, 타입체크 단계에서 “모듈을 찾을 수 없음”으로 루프가 깨지는 문제를 줄였다.

### 4. 남아 있던 앱 코드 에러 정리

그 다음 실제 앱 코드에서 남아 있던 미사용 변수, `display_name`/`nickname` 불일치, `E-2` 비자 문서 누락 같은 타입/린트 오류를 정리했다.

최종적으로 `ink-rn` 프로필 검증을 아래처럼 맞췄다.

```json
{
  "verify": ["npm test -- --watch=false", "npm run lint", "npm run typecheck"]
}
```

## 배운 점

- Ralph Loop는 검증 명령이 정확해야 효율이 나온다. 저장소 전체를 대충 검사하면 자동화가 아니라 잡음 수집기가 된다.
- React Native 앱과 Supabase Edge Functions는 실행 환경이 다르므로 타입체크와 린트 경계를 분리하는 게 맞다.
- `tsc` 실패를 모두 코드 문제로 보면 안 된다. 프로젝트 구조상 “어떤 파일을 누구 기준으로 검사할지”부터 설계해야 한다.
- 자동 수정 루프를 안정화하려면 기능 구현보다 먼저 **검증 파이프라인의 책임 범위**를 명확히 해야 한다.
